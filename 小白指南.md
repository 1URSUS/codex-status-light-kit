# Codex 状态灯小白指南

这份指南面向 Windows 用户，默认硬件是带 Micro USB 接口的 NodeMCU ESP8266/ESP-12F 开发板，软件是 Codex 桌面客户端的本地任务或 Codex CLI。

整个过程分成四段：先让 Windows 识别开发板，再烧录固件，然后用 Python 手动控制灯，最后接入 Codex hooks。每完成一段都先测试，不需要一次猜完所有问题。

## 需要准备什么

### 硬件

| 名称 | 数量 | 备注 |
| --- | ---: | --- |
| NodeMCU ESP8266 开发板 | 1 | ESP-12E/ESP-12F 均可，常见 CH340 或 CP2102 版本 |
| 红、黄、绿 LED | 各 1 | 也可以使用三色交通灯模组 |
| 220 欧姆电阻 | 3 | 普通 LED 每颗串联一个 |
| 面包板、杜邦线 | 若干 | 先免焊测试 |
| Micro USB 数据线 | 1 | 必须能传数据，只有充电功能的线不行 |

如果你使用交通灯模组，先确认它是高电平点亮还是低电平点亮。低电平点亮的模组需要把 `firmware/traffic_light_controller/pins_config.h` 中的 `LED_ACTIVE_LOW` 改成 `1`。

### 软件

| 软件 | 用途 |
| --- | --- |
| Arduino IDE 2.x | 安装 ESP8266 开发板支持并烧录固件 |
| Python 3.10+ | 运行 Codex hook 和串口程序 |
| Codex 桌面客户端或 Codex CLI | 产生状态事件 |

VS Code 中安装 Arduino 扩展后可以不用打开 Arduino IDE，但扩展背后仍然需要 Arduino CLI、ESP8266 Core 和正确的板型配置。第一次配置时 Arduino IDE 的报错更直观，因此本指南优先使用 Arduino IDE；固件烧录成功后，平时不需要保持 Arduino IDE 打开。

## 第 1 步：确认你用的是开发板

ESP-12F 可能指两种东西：

- 带 USB 接口、复位键、下载键和排针的完整开发板：可以按本指南操作。
- 只有金属屏蔽罩的小型裸 ESP-12F 模组：还需要稳定的 3.3V 供电、USB-TTL、电平与启动引脚电路，不建议初学者直接使用。

完整 ESP-12F 开发板在 Arduino IDE 中通常选择：

```text
NodeMCU 1.0 (ESP-12E Module)
```

板型名字写 ESP-12E 不影响常见 ESP-12F NodeMCU 开发板使用，两者在本项目所用引脚和 Arduino 配置上兼容。

## 第 2 步：接线

普通 LED 的长脚是正极，短脚是负极。每颗 LED 的长脚先串联一个 220 欧姆电阻，再接开发板；短脚接 GND。

| 灯 | NodeMCU 标记 | ESP8266 GPIO |
| --- | --- | ---: |
| 红灯 | `D1` | GPIO5 |
| 黄灯 | `D2` | GPIO4 |
| 绿灯 | `D7` | GPIO13 |
| 三个 LED 短脚 | `GND` | GND |

```text
D1 ---- 220R ---- 红 LED 长脚    红 LED 短脚 ---- GND
D2 ---- 220R ---- 黄 LED 长脚    黄 LED 短脚 ---- GND
D7 ---- 220R ---- 绿 LED 长脚    绿 LED 短脚 ---- GND
```

不要省略限流电阻，也不要把 LED 接到 5V。

## 第 3 步：让 Windows 识别开发板

先打开设备管理器，展开“端口 (COM 和 LPT)”，然后插拔开发板观察新增的设备。

正常名称通常类似：

```text
USB-SERIAL CH340 (COM7)
USB2.0-Serial (COM7)
Silicon Labs CP210x USB to UART Bridge (COM7)
```

“蓝牙链接上的标准串行 (COM3/COM4/COM5)”不是开发板。端口号本身不能证明驱动正常，必须看设备名称是否属于 CH340、CH341、CH343、CH910x、CP210x 或其他 USB 串口芯片。

### 查看 CH340 驱动是否安装

在 PowerShell 中运行：

```powershell
pnputil /enum-drivers | Select-String -Pattern "ch340|ch341|ch343|wch" -Context 2,4
```

看到 `ch341ser.inf`、`wch.cn` 和 `Ports`，说明驱动包安装过。驱动安装过不等于开发板已经连接成功，当前连接仍要以设备管理器新增真实 USB 串口为准。

如果插拔时设备管理器完全没有变化，按顺序排查：

1. 换一根明确支持数据传输的 USB 线。
2. 换电脑主板上的另一个 USB 口，不要先用 USB Hub。
3. 看“其他设备”或“通用串行总线控制器”中是否出现未知设备。
4. 从开发板卖家或 WCH/Silicon Labs 官方来源安装对应驱动。
5. 仍无变化时，检查开发板 USB 插座焊接，或换另一台电脑、另一块板交叉验证。

只有真实的 USB 串口出现后，才继续烧录。

## 第 4 步：配置 Arduino IDE

1. 安装并打开 [Arduino IDE](https://www.arduino.cc/en/software)。
2. 进入 `File > Preferences`。
3. 在 `Additional Boards Manager URLs` 中加入：

   ```text
   https://arduino.esp8266.com/stable/package_esp8266com_index.json
   ```

4. 打开 Boards Manager，搜索并安装 `esp8266`。
5. 打开 `firmware/traffic_light_controller/traffic_light_controller.ino`。
6. 在 Tools 菜单中选择下面的设置：

| 选项 | 建议值 |
| --- | --- |
| Board | `NodeMCU 1.0 (ESP-12E Module)` |
| Port | 设备管理器中真实的 USB 串口，例如 `COM7` |
| Upload Speed | `115200` |
| CPU Frequency | `80 MHz` |
| Flash Size | 常见 4MB 默认项即可 |

开发板背面常见提示可以这样理解：

| 背面提示 | 本项目如何处理 |
| --- | --- |
| 安装 CH340 驱动 | 需要，否则 Windows 无法正常得到 USB 串口 |
| 使用 9600bps | 多半是出厂程序的串口提示；本项目固件和 Python 通信使用 `115200` |
| 连接 WiFi | 本项目不需要 WiFi，只使用 USB 串口 |

先点击 Verify。当前固件不依赖 ArduinoJson，文件开头应是：

```cpp
#include "pins_config.h"
```

如果仍然报 `ArduinoJson.h: No such file or directory`，说明打开了旧文件。关闭当前草图，重新打开本仓库里的 `.ino`。

Verify 通过后点击 Upload。上传完成后，绿灯应开始闪烁，表示 `IDLE`。

### 上传时报端口错误

如果看到 `could not open port COMx`、`OSError 121` 或访问被拒绝：

1. 关闭 Arduino Serial Monitor 和 Serial Plotter。
2. 关闭 VS Code 串口监视器、串口助手和正在运行的 Python 状态灯程序。
3. 拔掉开发板，等待 5 秒再插回。
4. 回到设备管理器确认端口号是否变化。
5. 在 Arduino IDE 中重新选择 Port，Upload Speed 保持 `115200`。
6. 仍连不上时，开始 Upload 后按一下 `RST`；必要时按住 `FLASH`，开始写入后松开。
7. 最后再换数据线和 USB 口。

## 第 5 步：安装 Python 依赖

在仓库根目录打开 PowerShell：

```powershell
python --version
python -m pip install -r .\codex_hooks\requirements.txt
```

如果 `python` 命令不存在，安装 Python 3.10 或更高版本，安装时勾选 `Add Python to PATH`，然后重新打开 PowerShell。

## 第 6 步：先手动控制灯

列出 Python 看到的串口：

```powershell
python .\codex_hooks\send_signal.py --list-ports
```

如果自动识别不正确，固定开发板端口：

```powershell
setx STATUS_LIGHT_PORT COM7
```

把 `COM7` 换成自己的端口。`setx` 只对新启动的程序生效，执行后重新打开 PowerShell 和 Codex。

依次测试：

```powershell
python .\debug_tools\test_serial_command.py IDLE
python .\debug_tools\test_serial_command.py THINKING
python .\debug_tools\test_serial_command.py WAITING_USER
python .\debug_tools\test_serial_command.py TASK_COMPLETE
python .\debug_tools\test_serial_command.py TOOL_ERROR
```

| 命令 | 预期现象 |
| --- | --- |
| `IDLE` | 绿灯闪烁 |
| `THINKING` | 黄灯闪烁 |
| `WAITING_USER` | 红灯常亮，直到收到新状态 |
| `TASK_COMPLETE` | 绿灯常亮，约 60 秒后回到绿灯闪烁 |
| `TOOL_ERROR` | 红灯闪烁 |

如果灯效全部相反，把 `pins_config.h` 中的 `LED_ACTIVE_LOW` 改成 `1`，重新烧录。

## 第 7 步：安装 Codex hooks

在仓库根目录运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\install_windows.ps1 -InstallPythonDeps
```

脚本会修改：

```text
%USERPROFILE%\.codex\hooks.json
```

如果文件已经存在，脚本会：

1. 创建带时间戳的备份。
2. 保留其他项目已有的 hook。
3. 移除本项目旧的状态灯条目，加入当前版本，避免重复安装。

脚本不会自动批准或信任 hook。

## 第 8 步：让 Codex 信任 hook

Codex CLI：

1. 重启 Codex CLI 或开启新会话。
2. 输入 `/hooks`。
3. 找到包含 `send_signal.py` 的条目。
4. 检查路径正确后选择 trust/enable。

Codex 桌面客户端：

1. 使用本机 Local 环境，不要选择访问不到 USB 的云端环境。
2. 重启客户端或开启新任务。
3. 出现 hook 审查提示时，检查并信任 `send_signal.py`。

可以让 Codex 执行一个简单的本地读取任务来验证：提交提示后黄灯闪烁，任务完成后绿灯常亮。

## 状态与 Codex 事件

| Codex 事件 | 状态灯 |
| --- | --- |
| `SessionStart` | `IDLE` |
| `UserPromptSubmit` | `THINKING` |
| `PreToolUse` | `THINKING` |
| `PermissionRequest` | `WAITING_USER` |
| `PostToolUse` 失败 | `TOOL_ERROR` |
| `Stop` | `TASK_COMPLETE` |

Codex 对部分新执行机制的工具拦截可能不完整，但 `UserPromptSubmit` 和 `Stop` 仍能覆盖每个正常回合的开始与结束。具体事件以 [Codex Hooks 文档](https://developers.openai.com/codex/hooks)为准。

## 日志和模拟模式

默认日志位置：

```text
%LOCALAPPDATA%\codex-status-light-kit\logs\status_light.log
```

只测试 hook、不打开串口：

```powershell
setx STATUS_LIGHT_SIMULATE 1
```

关闭模拟模式：

```powershell
setx STATUS_LIGHT_SIMULATE 0
```

修改后要重新启动 Codex。

## 常见问题

### 手动测试成功，Codex 不触发

检查：

1. 是否重新启动了 Codex 或开启新任务。
2. CLI 中 `/hooks` 是否已经信任该 hook。
3. `%USERPROFILE%\.codex\hooks.json` 中的脚本绝对路径是否存在。
4. 是否误开了 `STATUS_LIGHT_SIMULATE=1`。
5. 日志中是否出现 `serial send failed` 或 `no supported USB serial port found`。

### Python 提示没有 serial

```powershell
python -m pip install pyserial
```

### 串口偶尔被占用

本项目会用跨进程锁和重试处理并行 hook，但 Arduino IDE 的串口监视器、其他串口软件仍会独占端口。运行 Codex 时保持这些工具关闭。

### 红灯一直亮

`WAITING_USER` 不会自动超时，因为 Codex 可能真的在等待你的批准。批准、拒绝或提交新提示后，后续 hook 会发送新的状态。

### 如何卸载

最稳妥的方式是恢复安装脚本生成的备份：

```text
%USERPROFILE%\.codex\hooks.json.bak-时间戳
```

也可以手动打开 `hooks.json`，删除 `statusMessage` 以 `Status light:` 开头的 hook 组。修改后重启 Codex。

## 最后检查清单

- 设备管理器显示真实 USB 串口，不是蓝牙 COM 口。
- Arduino IDE 使用 `NodeMCU 1.0 (ESP-12E Module)` 和 `115200` 上传速度。
- 固件 Verify、Upload 均成功，启动后绿灯闪烁。
- Python `--list-ports` 能看到开发板。
- 五种手动状态全部正常。
- Codex 已审查并信任 hook。
- 状态灯运行时没有其他程序占用串口。
